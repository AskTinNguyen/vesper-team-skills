# Example FAQ

## Customer Value
1. What painful behavior changes if this succeeds?
Routine follow-up and update collection moves from manual human chasing to supervised agent coordination.

2. Why will users trust this enough to adopt it?
Because every peer connection is explicit, every action is visible, and high-risk requests still require approval.

3. What is the fastest time-to-value for a new user?
A manager can connect one trusted peer and automate one recurring follow-up within the first day.

## Business Viability
4. What strategic priority does this advance?
It makes Vesper more valuable as a daily operating system for coordination, not just an isolated assistant.

5. What metric movement justifies continued investment?
Weekly active usage of coordination workflows and reduction in manually authored follow-up requests.

6. What happens if adoption is slower than expected?
Restrict scope to the highest-value use case: structured status collection for small teams.

## Technical Feasibility
7. What is the smallest viable technical slice?
Signed peer messaging plus a human review queue for inbound requests.

8. What dependencies could block delivery?
Permission boundaries, audit logging, and a clean event-ingress path.

9. What can we de-scope without harming the customer outcome?
File transfer, broad auto-execution, and organization-wide discovery.

## Risk and Governance
10. How could this fail users or erode trust?
If agents send requests users did not expect or if permissions are unclear.

11. What guardrails are mandatory before broad launch?
Explicit peer approval, rate limits, audit logs, and a one-click pause.

12. What kill criteria will stop this project?
If approval friction is too high for repeated use or if users report unclear control boundaries.

## Add More Hard Questions
- What happens if one side revokes trust mid-conversation?
- How do we prevent scope creep into remote tool execution?
- Are we solving a daily pain or just an impressive demo?

## What Changed After the FAQ?
- Scope changes: removed file transfer from v1.
- Sequencing changes: deliver signed messaging before proactive automation.
- Policy or guardrail changes: require explicit review for all task requests in beta.

## Why This Is Strong
- Includes multiple reasons the feature could fail.
- Produces real scope and sequencing changes.
- Keeps the initial release narrow and testable.
