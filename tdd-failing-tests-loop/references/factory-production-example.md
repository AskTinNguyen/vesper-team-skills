# Factory Production TDD Example

Use this reference as a worked example of the red-green-refactor loop.

## Request Shape

User request:
- review, overhaul, and refactor the Factory Production orchestration profile
- use TDD
- start from existing tests
- write failing tests first
- keep going until finished

## Step 1: Map The Contract

Read these surfaces first:
- `docs/internals/factory-production.md`
- lifecycle and watchdog tests
- Factory Production text, coordinator, policy, and lifecycle files
- orchestration profile instructions in `packages/shared/src/agent/orchestration-profiles.ts`

Extracted contract:
- checklist is the durable execution ledger
- memo is a scratchpad, not a parallel checklist
- planning gate should block only until real tasks and real memo notes exist
- Factory Production should explicitly support checklist-backed GitHub intake

## Step 2: Add Purposeful Red Tests

Add failing tests for:

1. Bootstrap metadata missing
   - fallback checklist seed items should carry structured bootstrap metadata

2. Planning gate too brittle
   - a user-authored item with a title that resembles an autogenerated placeholder should still count as real work

3. Policy and instructions underspecified
   - Factory Production instructions should mention `github_issue_sync`
   - policy should prioritize `github` ahead of team dispatch helpers

## Step 3: Confirm The Right Failures

Run the smallest affected suites.

Expected failures:
- seed item metadata undefined
- planning gate still using title heuristics
- profile instructions missing GitHub intake guidance
- policy missing `github` and `mission` family ordering

Do not implement fixes until the failures match the intended contract gaps.

## Step 4: Make Minimal Green Changes

Apply these kinds of fixes:
- add `bootstrap` metadata to Factory Production item metadata
- tag autogenerated fallback seed items with `metadata.factoryProduction.bootstrap = true`
- preserve seed metadata when initializing checklist runs
- replace title-prefix gate logic with structured placeholder detection
- update prompt/policy text to mention `github_issue_sync({ intake_mode: "factory-production" })`

## Step 5: Handle The Edge Case Revealed By Green

New edge found after the first refactor:
- editing an autogenerated placeholder into a real task still left stale bootstrap state behind

Add another failing test:
- an edited placeholder should count as real work

Fix:
- require both the bootstrap flag and an exact current placeholder-title match before treating an item as a bootstrap placeholder

## Step 6: Backfill Legacy Durable State

Protect older sessions by backfilling legacy two-item autogenerated checklists:
- only backfill when the checklist exactly matches the known legacy placeholder set
- avoid broad migrations that could relabel user-authored tasks

## Step 7: Sync Docs And Agent Instructions

Update:
- internal Factory Production doc
- orchestration profile instructions
- policy ordering

Keep tests covering the contract where practical.

## Step 8: Broaden Verification

Run in this order:
- focused Factory Production suites
- session manager/watchdog suites
- shared issue-sync/artifact/checklist suites
- typecheck

If a mixed test batch fails from unrelated harness setup, rerun the touched suites directly and note the harness issue instead of conflating it with the subsystem change.

## Final Report Shape

Report as:
- red test
- failure observed
- fix applied
- green result
- final verification commands
